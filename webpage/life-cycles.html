Software Development Life Cycle Model


A typical waterfall model


The Steps - Expanded

1) Requirements - This is where client specifications are communicated to the development team. Project managers and the client must discuss what the program must involve, and these specifications cannot be changed once they are agreed upon. Every aspect of the new system is to be discussed here, ranging from how users are going to input the information/interact with the program, and how data would preferably be stored. Once the requirements are specified, they are analysed. This analysis will discuss whether they can be incorporated into the final product and whether all requirements are valid.

Once completed, a specification is produced and used as guidelines throughout the development and testing processes.

2) Design - This is where the software design is produced to give a visual or logical representation of what the final product will potentially look like. This can be done through many means, but mind maps and pseudo code are a good way to visualise a process. This process also aids in identifying any hardware or system requirements, where the client may need to update software or hardware components if they are outdated. This process can also be used to discuss the system/software architecture, where the development team can identify how they wish to develop the program, i.e. via procedural programming or object-oriented programming.

3) Coding - This is the actual coding process, where the discussed specifications and design are translated into code. This stage is the focus of the development team, as it will dictate the quality of the final product. Testing can also be done at this stage gradually, to ensure snippets of code are functioning. However, the testing of the final product should be left until the testing stage.

4) Testing - After the coding process is complete, the program must be tested against the client specifications discussed in the requirement stage. This is done through a meticulous test plan which should cover much of the application and tests every aspect of the specification. This is largely to ensure the client is pleased with the final products, and no incidents may occur when incorporating the application into the actual system.

5) Maintenance - Once the application is tested and any bugs have been resolved, the program is then installed and setup on the machine. Once this is completed, the development team will continue to offer support for the application if any issues arise. This does not offer any new add-ons, which would require a new program/development process altogether.
